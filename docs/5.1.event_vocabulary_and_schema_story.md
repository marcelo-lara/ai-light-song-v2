# EPIC 5 - Story 5.1

## Event Vocabulary and Schema

## Goal

Define the canonical event vocabulary and strict event schema used by event inference, review tooling, and downstream lighting generation.

## Inputs

- the approved event vocabulary backlog in `docs/event_user_stories block 5.x.md`
- existing section, energy, harmonic, and symbolic artifacts that later stories will cite in evidence payloads
- downstream consumers that require stable event names and field shapes

## Dependencies

- a single canonical export name for each supported event type
- deterministic alias normalization for ingestion and review workflows
- a schema contract that distinguishes required fields from optional review and lighting fields

## Processing Steps

1. Define the canonical event list, descriptions, and categories.
2. Record allowed aliases for ingestion while preserving one canonical export name per event.
3. Define the event object with required fields for `id`, `type`, `start_time`, `end_time`, `confidence`, `intensity`, `evidence`, and `notes`.
4. Define evidence substructures that can point back to source layers, metrics, windows, and rule or model reasons.
5. Define optional fields for `section_name`, `source_layers`, `human_override`, `lighting_hints`, `candidates`, and other review metadata.
6. Add validation rules that reject unknown event names and malformed evidence objects.

## Output Artifacts

- `src/analyzer/contracts/event_vocabulary.json`
- `src/analyzer/contracts/song_event_schema.json`

## Acceptance Criteria

- A single source of truth exists for canonical event names, descriptions, categories, and aliases.
- Downstream components can validate that only canonical event names are exported.
- Required timing and confidence fields are unambiguous and always present in valid event payloads.
- Evidence objects can reference the feature layers and windows used for event decisions.

## Validation

- Confirm every exported event type appears in `event_vocabulary.json`.
- Confirm schema validation fails for unknown event names, missing timing fields, or malformed evidence payloads.
- Confirm alias normalization never changes the canonical exported event name unexpectedly.

## Failure Cases

- Multiple names are allowed to leak into exports for the same event concept.
- Schema rules allow events without confidence, timing, or evidence provenance.
- Review tools must guess field meanings because the contract is underspecified.

## Out of Scope

- Per-song event inference
- Genre-specific threshold tuning
- Fixture-specific lighting logic
