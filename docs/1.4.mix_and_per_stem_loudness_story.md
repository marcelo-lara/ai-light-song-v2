# EPIC 1 - Story 1.4

## Mix and Per-Stem Loudness (RMS and Envelope)

## Goal

Extract debugger-facing loudness profiles from the source mix and all required Story 1.1 stems so the helper UI can compare fast RMS motion against a slower loudness envelope without creating new routine files under `data/output/`.

## Inputs

- Source song from `data/songs/<Song - Artist>.mp3`
- Stem outputs from Story 1.1:
  - `data/stems/<Song - Artist>/bass.wav`
  - `data/stems/<Song - Artist>/drums.wav`
  - `data/stems/<Song - Artist>/harmonic.wav`
  - `data/stems/<Song - Artist>/vocals.wav`

## Dependencies

- Essentia audio loading
- NumPy for RMS calculation and normalization
- Story 1.1 stem-generation metadata and cached stem files

## Processing Steps

1. Load the source mix and all required stems as mono at the canonical analysis sample rate.
2. Use the mix duration as the shared time base for all loudness outputs.
3. Compute RMS loudness on fixed 10 ms windows for the mix and every stem.
4. Compute a slower loudness envelope on fixed 200 ms windows for the mix and every stem.
5. Normalize each source independently to a stable `0.0` to `1.0` display range while preserving the raw RMS values in the artifact.
6. Persist both artifacts under the Essentia producer namespace.
7. Record both artifact paths in `data/output/<Song - Artist>/info.json` so the helper UI can discover them directly from the artifact tree.
8. Render two helper UI lanes:
   - one stacked lane for per-source RMS loudness
   - one stacked lane for per-source loudness envelope

## Output Artifacts

- `data/artifacts/<Song - Artist>/essentia/rms_loudness.json`
- `data/artifacts/<Song - Artist>/essentia/loudness_envelope.json`

Generated loudness artifacts must remain under `data/artifacts/<Song - Artist>/`. Do not add new routine files under `data/output/<Song - Artist>/` for this story.

## Acceptance Criteria

- The source mix and all four required Story 1.1 stems are included in both artifacts.
- `rms_loudness.json` uses 10 ms windows on a shared song timeline.
- `loudness_envelope.json` uses 200 ms windows on the same shared timeline.
- Every frame carries one raw value and one normalized value per source in stable source order.
- The helper UI exposes one per-source RMS lane and one per-source envelope lane.
- No new routine file is added under `data/output/<Song - Artist>/`.

## Validation

- Confirm both artifact files exist and are non-empty.
- Confirm `sources` contains exactly `mix`, `bass`, `drums`, `harmonic`, and `vocals` in stable order.
- Confirm `metadata.interval_ms` is `10` for RMS output.
- Confirm `metadata.window_ms` is `200` for envelope output.
- Confirm every frame carries five raw values and five normalized values.
- Spot-check that silent or quiet passages produce lower values than active passages for the same source.
- Confirm the helper UI loads both artifacts without breaking songs that do not yet have backfilled loudness files.

## Failure Cases

- Missing source mix file
- Missing or unreadable Story 1.1 stem file
- Essentia import failure
- Empty or unreadable audio input
- Per-frame source arrays with inconsistent lengths
- Non-monotonic frame timestamps

## Out of Scope

- LUFS or psychoacoustic loudness calibration
- Beat-aligned loudness summaries for downstream event inference
- Automatic thresholding or event extraction from the new loudness artifacts
- Writing additional consumer-facing loudness files under `data/output/`
