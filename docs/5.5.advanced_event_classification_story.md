# EPIC 5 - Story 5.5

## Advanced Musical Event Classification

## Goal

Refine baseline event candidates into higher-level musical event types that matter for show design, including drop variants, vocal-driven moments, rhythmic textures, and reset mechanics.

## Inputs

- event vocabulary and schema from Story 5.1
- event threshold profiles from `event_threshold_profiles.json`
- normalized event feature layer from Story 5.2
- rule candidates from Story 5.3
- named identifier hints from Story 5.4
- section windows from Story 4.2
- vocal and symbolic context from `layer_b_symbolic.json` when available
- optional word- or line-timed lyric clues from `data/reference/<Song - Artist>/moises/lyrics.json` when available
- spectral evidence from `essentia/fft_bands.json`
- per-stem loudness from `essentia/rms_loudness.json`

## Dependencies

- deterministic parent-to-subtype fallback logic
- explicit evidence summaries for every refined label
- stable handling of ambiguous cases where multiple event types score similarly

## Processing Steps

1. **Drop Refinement with Spectral Validation:** Refine generic `drop` candidates into subtypes by measuring the delta in the 60-150Hz ('Bass') FFT band and the broadband RMS jump. An `explode` variant *must* show a sharp spike across all 7 FFT bands simultaneously.
2. **Vocal Event Verification:** Detect vocal-driven events using vocal stem RMS dominance. A `vocal_spotlight` MUST be snapped to the exact millisecond the Vocal Stem RMS exceeds the noise floor. If the instrumentation drops out simultaneously, the event priority is upgraded.
3. Detect reset mechanics and sound-context states such as `energy_reset`, `layer_remove`, `layer_add`, `percussion_break`, and `instrumental_bed` using abrupt energy removal, stem dominance, spectral simplification, and later reintroduction cues.
4. **Rhythmic Texture Inference:** Identify patterns like `heartbeat_pattern` or `four_on_the_floor` by analyzing the periodicity of transients in the Drum Stem RMS envelope.
5. **The Physical Verification Gate:** For every refined event, cross-reference the candidate window against `fft_bands.json` and `rms_loudness.json`. Both **start_time** and **end_time** MUST be snapped to source-specific RMS envelope crossings (e.g., Drum RMS for rhythmic textures, Vocal RMS for spotlight moments). This eliminates "shifted" timing caused by window-averaging or symbolic-only guesses.
6. Fall back to the parent event label when subtype evidence is insufficient, preserving the competing candidates for review.
7. Attach intensity, confidence, section references, and concise notes that summarize why the refined label was chosen.

## Output Artifacts

- `data/artifacts/<Song - Artist>/event_inference/events.machine.json`

## Acceptance Criteria

- Generic event candidates can be refined into more specific musical types when the evidence is measurable and strong enough.
- Lower-confidence cases fall back to a stable parent label instead of forcing a brittle subtype.
- Vocal-driven and reset events remain explainable even when lyric transcription is incomplete.
- Stem-aware sound-context events remain explainable even when note transcription is sparse or beat-centric features alone would miss a voice-led or percussion-led moment.
- Exported evidence distinguishes explosive, groove-based, punchy, or restrained releases in measurable ways.
- All rhythmic textures and high-intensity transitions are physically validated against spectral transients to ensure zero-latency synchronization with the lighting rig.

## Validation

- Compare refined labels against reviewed benchmark songs and expected subtype behavior.
- Confirm ambiguous subtype cases preserve alternative candidates for later review.
- Confirm repeated runs on the same song emit the same subtype decisions and evidence summaries.

## Failure Cases

- The classifier overuses specific subtypes without sufficient evidence.
- Vocal-driven events require exact lyrics and fail when only stem activity is available.
- Reset events are reduced to vague prose instead of traceable feature changes.

## Out of Scope

- Human review UI implementation
- Fixture-specific lighting decisions
- Training a learned classifier
