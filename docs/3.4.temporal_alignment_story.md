# EPIC 3 - Story 3.4
## [PENDING] Temporal Alignment and Anti-Stutter Audit

## Goal

Align symbolic note events and derived symbolic features to the canonical beat and bar grid so all later artifacts share one timing model.

This timing model must also support exact phrase and callback anchors for downstream lighting-score generation.
It must also implement **anti-stutter buffering logic** to ensure that "Audit" data stays sample-accurate with the AI prediction data.

## Inputs

- Note events from Story 2.4
- Source-level symbolic validation output from Story 2.4 when available
- Canonical beat and bar grid from Story 1.2
- Section windows from Story 3.1

## Dependencies

- Snap-to-grid heuristics and tolerance thresholds

## Processing Steps

1. **Anti-Stutter Buffering**:
   - Implement precise look-ahead and history buffering to ensure that the FFT "Enforcer" audit doesn't drift from the beat-grid timing.
   - Align features exactly to the 100ms frame boundaries while maintaining sample-accurate references to the raw audio transients.
2. **Temporal Grid Matching**:
1. Match note onsets to the nearest valid beat when beat-aligned outputs are required.
2. Preserve raw onset times alongside snapped alignment fields when the story needs both.
3. Assign each note event to a 1-indexed bar and beat-in-bar index.
4. Attach section identifiers using section windows.
5. Derive phrase-anchor or phrase-window timing references when the symbolic layer exposes phrase-level summaries.
6. Use the aligned outputs as the only valid source for later bar-level, section-level, and phrase-level symbolic summaries.

## Alignment Rules

- Preserve original `time` and `end_s`.
- Add alignment fields rather than overwriting raw timings.
- Use deterministic nearest-neighbor snapping.
- Use `aligned_beat` for beat-in-bar and a separate field such as `aligned_beat_global` when absolute beat numbering is needed.
- If an onset is outside the allowed tolerance, keep the raw time and mark the alignment as unresolved instead of forcing a bad snap.

For downstream lighting use, phrase-anchor field names should be normalized as:

- `phrase_windows[].id`
- `phrase_windows[].phrase_group_id`
- `phrase_windows[].start_s`
- `phrase_windows[].end_s`
- `phrase_windows[].start_bar`
- `phrase_windows[].end_bar`
- `phrase_windows[].start_beat`
- `phrase_windows[].end_beat`
- `phrase_windows[].section_id`

## Acceptance Criteria

- Aligned events are consistent with the canonical beat grid.
- No second, conflicting timing model is introduced.
- Phrase-anchor timing remains compatible with exact downstream cue placement.
- Unresolved alignments are rare and explicitly marked.

## Validation

- Spot-check aligned events near section boundaries.
- Confirm bar numbering is 1-indexed and monotonic.
- Confirm phrase windows and anchors line up with the aligned note and bar structure.
- Confirm density-per-bar output uses the aligned bar IDs rather than inferred ranges.

## Failure Cases

- Forced snapping across large timing offsets
- Section assignment lags behind actual section windows
- Phrase anchors drift away from the aligned note material they are meant to summarize
- Alignment fields overwrite raw timing data

## Out of Scope

- Human correction UI for alignment
- Swing-aware or expressive timing transcription models
