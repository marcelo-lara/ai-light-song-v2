# EPIC 5 - Story 5.1

## Find Chord Patterns

## Goal

Extract repeated multi-bar chord progressions from beat-level chord annotations and publish Layer D pattern summaries that downstream stories can consume without re-deriving harmonic repetition.

This story does not replace motif-level or phrase-level repetition summaries derived from note events. Those remain part of `layer_b_symbolic.json` under EPIC 3.

## Purpose

`find_chord_patterns(...)` turns a flat list of beat rows into a compact summary of recurring harmonic patterns.

It is designed for analyzer output where each beat may carry a bar number, beat number, time, and chord label.

Its scope is harmonic loop and progression structure, not melodic-motif discovery.

The pure function returns `None` when there are not enough valid bars or when no repeated pattern is found. Story-level orchestration must handle that outcome explicitly and must not invent fallback patterns.

## Expected Input

The main input is `beats: list[dict[str, Any]]`.

Each usable row is expected to contain:

- `bar`: 1-based bar index
- `beat`: 1-based beat index within a 4-beat bar
- `time`: beat timestamp in seconds
- `chord`: chord label such as `C`, `Am`, `Fmaj7`, `G7`, or `N`

Rows that are not dictionaries, fall outside bars `>= 1`, or use beats outside `1..4` are ignored.

## Inputs

- beat-level chord annotations aligned to the canonical Story 1.2 beat grid
- chord labels derived from harmonic analysis and attached to the beat timeline
- optional source-path metadata for the beat file used to generate the pattern artifact

## Dependencies

- deterministic beat grouping into 4-beat bars
- deterministic chord normalization before comparison
- deterministic candidate scoring and overlap rejection
- standard-library-only implementation unless a later repo contract explicitly changes that requirement

## High-Level Flow

1. `_build_bars` groups beat rows into bars with exactly four beat slots.
2. `_normalize_chord` reduces chord spelling noise so similar harmonies compare cleanly.
3. `_best_candidate` searches for the strongest repeating bar window.
4. Each accepted match marks its bars as covered, so later matches cannot overlap it.
5. `_pattern_payload` converts the internal match into a stable JSON-friendly structure.

This repeats until no more valid repeating windows can be found.

## Matching Rules

- Only windows of 2 to 8 bars are considered.
- 4-bar patterns are preferred over all other lengths.
- 1-bar loops are intentionally ignored.
- 2-bar patterns require exact beat-by-beat matches.
- Patterns longer than 2 bars may differ by up to 3 beats total.
- A candidate must occur at least twice to be kept.
- Overlapping matches are rejected through the `covered` bar index set.

## Chord Normalization

`_normalize_chord` strips some seventh-chord detail before comparison:

- `maj7` and `M7` become `maj`
- `min7` and `m7` become `m`
- plain `7` is removed
- `N` becomes `None`

That means labels like `Cmaj7` and `Cmaj` compare as the same harmonic unit for pattern detection.

## Candidate Scoring

`_group_candidate` evaluates one starting window and finds all non-overlapping occurrences that resemble it.

The score favors:

- more total bars covered
- more repetition
- longer windows
- 4-bar windows specifically

The score is reduced by accumulated beat mismatches. The first bar length that produces at least one viable candidate wins, so 4-bar windows are exhausted before shorter or longer alternatives.

## Representative Sequence

When a pattern has multiple occurrences, `_representative_window` builds a consensus version of the progression:

- each beat position votes across all occurrences
- the most common chord label wins
- ties fall back to the occurrence with the fewest mismatches

This representative window is then rendered as a compact display string such as:

```text
G#-A#|Cm|G#-A#|Cm
```

Within one bar, repeated adjacent beat labels are collapsed by `_display_bar`, and bars are joined with `|` by `_display_window`.

## Output Artifacts

- `data/artifacts/<Song - Artist>/pattern_mining/chord_patterns.json`: producer-scoped result from `find_chord_patterns(...)`
- `data/artifacts/<Song - Artist>/layer_d_patterns.json`: canonical Layer D artifact consumed by Story 5.2

Layer D complements symbolic repetition summaries rather than replacing them. Repeated motifs, contour repetition, and phrase-level note-pattern summaries remain owned by `layer_b_symbolic.json`.

For downstream lighting use, the canonical pattern-reference fields are:

- `patterns[].id`
- `patterns[].sequence`
- `patterns[].occurrences[]`
- `patterns[].occurrences[].start_s`
- `patterns[].occurrences[].end_s`
- `patterns[].occurrences[].start_bar`
- `patterns[].occurrences[].end_bar`

If `find_chord_patterns(...)` returns `None`, the story wrapper must preserve that outcome explicitly by writing a stable zero-pattern artifact or by documenting and enforcing a no-file contract across downstream consumers. It must not synthesize a pattern to satisfy the schema.

Successful results look like this at a high level:

## Acceptance Criteria

- `find_chord_patterns(...)` ignores malformed beat rows instead of crashing on them.
- Only 2-bar to 8-bar windows are evaluated, with 4-bar windows preferred first.
- 2-bar matches are exact and longer matches allow at most 3 beat mismatches total.
- Overlapping matches are rejected through covered-bar tracking.
- The emitted pattern payload is deterministic for the same beat input.
- Story 5.2 can consume the Layer D artifact without re-running pattern discovery.

## Validation

- Confirm the implementation emits the documented schema shape for `layer_d_patterns.json` on representative songs.
- Confirm 1-bar loops are never emitted.
- Confirm normalization merges seventh-chord spelling noise as specified without collapsing unrelated roots.
- Confirm the representative sequence matches the consensus of the retained occurrences.
- Confirm no bars are reused across overlapping matches.

## Failure Cases

- non-dictionary or malformed beat rows cause runtime failure instead of being ignored
- overlapping windows are emitted as separate accepted occurrences
- chord normalization merges unrelated harmonic roots
- the wrapper invents fallback patterns when the pure function returns `None`

## Out of Scope

- non-4/4 bar models
- music-theory-perfect chord equivalence
- stem-aware or melody-aware pattern mining
