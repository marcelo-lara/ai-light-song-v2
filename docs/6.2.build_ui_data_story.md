# EPIC 6 - Story 6.2 [PENDING]

## Build UI Data

## Goal

Project the canonical analysis artifacts into compact UI-facing output files under `data/output/<Song - Artist>/` without exposing producer-specific artifact schemas to downstream consumers.

This story defines presentation-oriented output contracts for beat and section timelines. The schemas are anchored to the existing sample files in `data/output/What a Feeling - Courtney Storm/`, but values must be derived from current generated artifacts rather than copied from sample data or validation-only references.

`data/output/<Song - Artist>/` is a stable UI contract. The required files are `beats.json`, `hints.json`, `info.json`, `sections.json`, `song_event_timeline.json`, and `lighting_score.md`. Do not add or remove files there unless a UI contract change makes that strictly required.

## Inputs

- `data/artifacts/<Song - Artist>/essentia/beats.json`
- `data/artifacts/<Song - Artist>/layer_a_harmonic.json`
- `data/artifacts/<Song - Artist>/symbolic_transcription/basic_pitch/bass.json`
- `data/artifacts/<Song - Artist>/section_segmentation/sections.json`

## Dependencies

- Stable canonical beat timing from Story 1.2
- Stable section windows from Story 4.2
- Stable chord-event timing from EPIC 2 harmonic analysis
- Stable bass note cache from Story 3.1

## Processing Steps

1. Load the existing artifact JSON files produced by the upstream timing, harmonic, and section stages.
2. Load the raw bass note cache from `symbolic_transcription/basic_pitch/bass.json`.
3. Convert Basic Pitch MIDI pitches into pitch-class note names and align each bass-note onset to the nearest existing beat time without creating extra beat rows.
4. Build `data/output/<Song - Artist>/beats.json` by projecting beat time, beat-in-bar, bar number, active chord label, beat-aligned bass note, and beat type into a compact UI row shape.
5. Assign each beat row the active chord whose `[time, end_s)` window contains the beat timestamp, falling back to the most recent prior chord when a beat lands on a boundary.
6. Populate `bass` with the deterministic beat-aligned note name from the Basic Pitch bass source when available, otherwise leave it `null`.
7. Build `data/output/<Song - Artist>/sections.json` by projecting section start, end, and label into a compact UI row shape.
8. Normalize section labels for presentation without changing the underlying section boundaries, embedding the numeric section id prefix and rounded confidence suffix into the UI-facing `label` text.
9. Emit explicit placeholder values for fields that do not yet have an authoritative upstream source in this UI projection: `description: ""` and `hints: []`.
10. Record the generated output paths in `data/output/<Song - Artist>/info.json`.

## Output Artifacts

- `data/output/<Song - Artist>/beats.json`
- `data/output/<Song - Artist>/sections.json`

Note: `data/output/<Song - Artist>/hints.json` is a separate later contract and is the authoritative editable hint store. The `hints` array in `sections.json` remains a placeholder for compatibility.

## Output Schemas

`beats.json`

```json

  {

    "beat": 2,
    "bar": 2,
    "bass": "C#",
    "chord": "D#",

```

`sections.json`

```json
  {
    "start": 17.62,
    "end": 35.34,
    "label": "003 Verse (0.76)",
    "description": "",
    "hints": []
  }
]
```

## Acceptance Criteria

- The generated files live under `data/output/<Song - Artist>/`, not under `data/artifacts/<Song - Artist>/`.
- `beats.json` contains only `time`, `beat`, `bar`, `bass`, `chord`, and `type`.
- `sections.json` contains only `start`, `end`, `label`, `description`, and `hints`.
- This story preserves the stable UI output whitelist and does not introduce additional per-song files under `data/output/<Song - Artist>/`.
- The UI-facing `label` embeds the numeric section id prefix and a confidence value rounded to 2 decimals, for example `001 Intro (0.74)`.
- Output values are deterministic projections from the current artifacts, not copies from sample files or `/data/reference/`.
- `bass` is derived from the Basic Pitch bass cache as a pitch-class note name and aligned onto the existing beat grid without creating or deleting beats.
- `description` remains an explicit default field until a later story defines an authoritative source for it.
- `hints` remains an explicit placeholder field in `sections.json` and is not the authoritative editable hint store.
- Section generation does not rerun segmentation logic; it only projects the existing section artifact into the UI contract.

## Validation

- Confirm the upstream artifact files exist before output generation.
- Confirm beat times match the canonical timing artifact after UI rounding.
- Confirm beat-to-chord assignment is deterministic for repeated runs on the same song.
- Confirm beat-aligned bass-note assignment is deterministic for repeated runs on the same song.
- Confirm section windows match the existing section-segmentation artifact after UI rounding.
- Confirm the generated files match the documented row shapes exactly.

## Failure Cases

- The implementation re-infers beats, chords, or sections instead of projecting existing artifacts.
- The UI output silently reads from `/data/reference/` when an artifact field is missing.
- Beat rows expose producer-specific fields such as `index`, `confidence`, `pitch`, or `beat_in_bar` instead of the compact UI contract.
- Section output omits required placeholder fields or fills them with invented prose.

## Out of Scope

- Feature-to-lighting mapping
- Fixture-aware orchestration
- Heuristic enrichment of `bass`, `description`, or `hints`
