# EPIC 5 - Story 5.3
## 1D-CNN Event Classifier and Interpretability Pipeline

## Goal
Implement a **Temporal 1D-Convolutional Neural Network (1D-CNN)** to classify musical events from the multi-modal feature streams produced in Story 4.4. Unlike rule-based systems, this model will recognize complex, time-dependent "shapes" in the music—such as the specific energy/density trajectory of a "Drop" or "Build-up"—across different genres.

## Inputs
- **Contextual Feature Vectors (from Story 4.4)**: Z-score normalized streams of loudness, rhythmic density, spectral flux, and harmonic tension.
- **Ground Truth Labels (from Story 5.5)**: Human-annotated benchmark files (`/benchmark_annotations/`) mapped to the Story 5.1 Event Vocabulary.
- **Rule-Based Candidates (from Story 5.2)**: Initial event guesses to be used as a "baseline" or as additional features for a hybrid approach.

## 1D-CNN Architecture Specification
The model shall follow a "sliding window" architecture to capture temporal motifs:

1.  **Input Layer**: Accepts a window of $N$ time-steps (e.g., 256 steps at 100ms resolution $ pprox$ 25 seconds) by $M$ features (energy, density, etc.).
2.  **Temporal Convolutional Blocks**:
    - **Layers**: 3–5 layers of 1D-Convolution.
    - **Kernel Sizes**: Variable sizes (e.g., 3, 5, and 9) to capture both micro-transients (drum fills) and macro-transitions (chorus entries).
    - **Activation**: ReLU or LeakyReLU.
3.  **Max-Pooling Layers**: Downsample the temporal dimension to provide local translation invariance (the exact millisecond of an event matters less than its general position).
4.  **Global Average Pooling**: Compress the final feature maps into a single vector representing the entire window's "musical character."
5.  **Dense Classifier Head**:
    - A multi-label output layer with Sigmoid activation.
    - Each node represents one event type from the **Event Vocabulary** (e.g., `is_drop`, `is_buildup`, `is_solo`).

## Processing Steps
1.  **Dataset Construction**:
    - Slice benchmark songs into overlapping windows.
    - Augment the data by slightly shifting window boundaries and adding small amounts of noise to the feature vectors to improve robustness.
2.  **Training Loop**:
    - Use Binary Cross-Entropy (BCE) loss to support multi-label classification.
    - Implement a custom "Temporal Weighted Loss" to penalize missing a "Drop" more heavily than missing a minor "Drum Fill."
3.  **Explainability (Saliency Maps)**:
    - Use Gradient-weighted Class Activation Mapping (Grad-CAM) to identify which feature (e.g., "High-Frequency Energy") and which specific time-window caused the model to trigger an event.
4.  **Inference & Smoothing**:
    - Load and run the promoted weights from `models/event_classifier/1d_cnn_v1.pth` across the full song.
    - Apply a "Softmax-Smoothing" or Hysteresis filter to prevent the classifier from flickering between labels.

## Output Artifacts
- `models/event_classifier/1d_cnn_v1.pth` (or `.h5`): The trained model weights.
- `data/artifacts/<Song>/event_inference/events.ml.json`: The predicted event timeline with confidence scores.
- `data/artifacts/<Song>/event_inference/saliency_explanations.json`: Data mapping labels back to the contributing input features for the UI "Insight" panel.

## Acceptance Criteria
- **Temporal Sensitivity**: The model must detect a "Drop" within ±200ms of the human-annotated timestamp.
- **Confidence Calibration**: Events labeled with >90% confidence should have a precision of at least 85% compared to benchmarks.
- **Explainability**: For every "Drop" detected, the system must be able to export the top 3 contributing features from the Story 4.4 vector.

## Validation
- Compare the 1D-CNN F1-score against the Story 5.2 "Hard Rules" baseline.
- Verify that the model's performance does not degrade on "Out-of-Distribution" genres not present in the initial training set (Generalization Test).
- Verify inference consumes the existing promoted weights artifact and does not trigger a new training run when `models/event_classifier/1d_cnn_v1.pth` is already present.

# EPIC 5 - Story 5.3
## 1D-CNN Training and Dataset Generation

## Goal
Create a repeatable training pipeline to produce the weights for the 1D-CNN Event Classifier. This story converts reference `human_hints.json` windows and the 100ms contextual feature grid into a supervised multi-label dataset the neural network can learn from.

Training is conditional: the training routine should run only when `models/event_classifier/1d_cnn_v1.pth` does not already exist. If the weights file is already present, the pipeline should treat it as the current promoted artifact and skip retraining unless an explicit retraining workflow is requested.

## Inputs
- **Feature Streams**: `data/artifacts/<Song>/event_inference/contextual_features.json` (the 100ms "X" data produced by Story 4.4).
- **Human Truth**: `data/reference/<Song>/human/human_hints.json` (The "Y" labels).
- **Model Spec**: 1D-CNN Architecture defined in Story 5.3.

## Processing Steps
1. **Label Vectorization**:
    - Map `human_hints.json` event types (e.g., "Drop", "Build") to indices in a multi-hot vector based on the Story 5.1 Event Vocabulary.
    - For every 100ms feature frame, assign a 1 or 0 for each event class based on whether the timestamp falls within a human-annotated window.
2. **Windowed Dataset Creation**:
    - Implement a sliding window (256 steps @ 100ms resolution).
    - Sample windows containing events more frequently during training to handle class imbalance (since music is mostly "no event").
3. **Training Execution**:
    - Check whether `models/event_classifier/1d_cnn_v1.pth` already exists before starting training.
    - Skip training when that promoted weights file already exists, leaving the current artifact in place.
    - Split songs into Training (80%) and Validation (20%) sets.
    - Train using Binary Cross-Entropy (BCE) Loss with the Adam optimizer.
4. **Weight Export**:
    - Save the best-performing model (based on Validation F1-score, with precision gate enforcement) to `models/event_classifier/1d_cnn_v1.pth`.

## Output Artifacts
- `models/event_classifier/1d_cnn_v1.pth`: The serialized weights.
- `models/event_classifier/metadata.json`: Records which songs were used for training and the final accuracy metrics.

## Acceptance Criteria
- **Training Success**: The loss curve must show convergence over at least 20 epochs.
- **Validation**: The model must achieve a precision of >70% on the validation set before being promoted to `v1`.
- **Cinderella Test**: After training on a dataset including "Cinderella - Ella Lee" (or similar tracks), the model must detect the major energy shifts defined in its `human_hints.json`.
- **Skip Behavior**: If `models/event_classifier/1d_cnn_v1.pth` already exists, the default training path must skip retraining and preserve the existing promoted weights.

## Validation
- Run `python -m scripts.train_event_classifier --songs-root /data/reference --artifacts-root /data/artifacts`.
- When the weights file is absent, verify the `.pth` file is created and contains non-zero weights.
- When the weights file already exists, verify the default training path skips retraining rather than overwriting the promoted artifact.

## Implementation Notes

- The canonical training entrypoint is `python -m scripts.train_event_classifier`.
- The trainer consumes `contextual_features.json`, derives multi-hot labels from `human_hints.json`, and writes both weights and metrics under `models/event_classifier/`.
- Promotion to `1d_cnn_v1.pth` is gated on the validation precision threshold. Failed promotion still writes `metadata.json` so the result can be inspected without pretending the model passed Story 5.3 acceptance.
- The default Story 5.3 contract is artifact-aware: if `models/event_classifier/1d_cnn_v1.pth` already exists, training should not rerun unless an explicit retraining workflow is invoked.
