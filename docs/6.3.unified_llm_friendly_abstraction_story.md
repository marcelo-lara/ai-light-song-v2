# EPIC 6 - Story 6.3
## LLM-Friendly Song Map Abstraction

## Goal
Consolidate the stable **Musical Signatures (4.6)** and the **Structural Segments (3.2)** into a unified "Song Map." This acts as the final "handshake" between the Perception Layer and the AI Light Show Designer (LLM).

## Inputs
- **Musical Signatures** (from Story 6.3): Stable neural energy and tension profiles.
- **Section Segmentation** (from Story 3.1): Validated timestamps for Verse, Chorus, Bridge, etc.
- **Loudness & FFT Trends** (from Story 1.3/1.4): High-level dynamic peaks.

## Processing Steps
1.  **Contextual Labeling**: For each segment defined in 3.2, look at the signature values in 4.6. Assign human labels like:
    - *"Energy 0.8, Rhythmic Clarity 0.9" → "Driving High-Energy Chorus"*
    - *"Energy 0.2, Harmonic Tension 0.7" → "Atmospheric Tense Bridge"*
2.  **Event Prediction Buffering**: Identify "Event Candidates" from Story 5.2 to give the LLM hints about where "The Drop" or a "Solo" might occur.
3.  **Token-Optimized Mapping**: Flatten this data into a JSON/Text format designed for a prompt window.

## Output Artifacts
- `data/artifacts/<Song>/llm_abstraction/song_map.json`: The source of truth for the AI Agent.

## Acceptance Criteria
- **Dependency Integrity**: The abstraction MUST include validated section names from 3.2.
- **Production Quality**: No "flickering" descriptions; labels must remain constant throughout a defined section.

# EPIC 6 - Story 6.3

## LLM-Friendly Abstraction

## Goal

Convert symbolic features into concise, musician-readable descriptors that are useful to both humans and downstream prompt-based workflows.

## Inputs

- Symbolic summaries from Story 4.3
- Aligned phrase and section information from Story 3.4

## Dependencies

- Deterministic descriptor templates
- Optional later LLM refinement, but not required for baseline generation

## Processing Steps

1. Convert core symbolic metrics into a controlled vocabulary.
2. Build short global and phrase-level descriptions using deterministic templates.
3. Avoid vague language that hides the actual feature values.
4. Persist both structured metrics and human-readable descriptions.

## Output Schema

```json
{
  "description": "Repeated mid-register pattern with rising contour, moderate density, and leaping bass movement.",
  "abstraction": {
    "core_terms": {
      "texture": "layered",
      "register_label": "mid",
      "melodic_contour": "rising",
      "bass_motion": "leaping",
      "repetition_level": "strong repetition",
      "sustain_profile": "mixed sustain"
    },
    "section_descriptions": [
      {
        "section_id": "section-001",
        "description": "Intro spans bars 1-8 with a layered texture, rising contour, moderate activity, and occasional repetition."
      }
    ],
    "phrase_group_descriptions": [
      {
        "phrase_group_id": "phrase_group_A",
        "motif_group_id": "motif_alpha",
        "description": "Phrase Group A recurs 3 times with rising contour and active note motion."
      }
    ]
  }
}
```

## Acceptance Criteria

- The description is understandable by a musician.
- The wording reflects the underlying metrics rather than generic prose.
- Repeated runs on the same artifact yield the same text unless upstream features change.
- Section and repeated-phrase descriptions are derived from the canonical structured fields rather than a second independent analysis path.

## Validation

- Compare the text against the structured symbolic summary.
- Confirm the description does not claim features missing from the actual metrics.

## Failure Cases

- Overly generic text that fits every song
- Descriptions contradict the structured symbolic data
- Abstraction layer becomes dependent on an external LLM to function

## Out of Scope

- Creative lyric interpretation
- Genre storytelling beyond the symbolic descriptor set
