# EPIC 3 - Story 3.3

## Temporal Alignment

## Goal

Align symbolic note events and derived symbolic features to the canonical beat and bar grid so all later artifacts share one timing model.

## Inputs

- Note events from Story 3.1
- Canonical beat and bar grid from Story 1.2
- Section windows from Story 4.2

## Dependencies

- Snap-to-grid heuristics and tolerance thresholds

## Processing Steps

1. Match note onsets to the nearest valid beat when beat-aligned outputs are required.
2. Preserve raw onset times alongside snapped alignment fields when the story needs both.
3. Assign each note event to a bar and beat index.
4. Attach section identifiers using section windows.
5. Use the aligned outputs as the only valid source for later bar-level and section-level symbolic summaries.

## Alignment Rules

- Preserve original `time` and `end_s`.
- Add alignment fields rather than overwriting raw timings.
- Use deterministic nearest-neighbor snapping.
- If an onset is outside the allowed tolerance, keep the raw time and mark the alignment as unresolved instead of forcing a bad snap.

## Output Schema

```json
{
  "note_events": [
    {
      "time": 1.230,
      "aligned_beat": 3,
      "aligned_bar": 1,
      "beat_time_delta": 0.018,
      "section_name": "Intro"
    }
  ]
}
```

## Acceptance Criteria

- Aligned events are consistent with the canonical beat grid.
- No second, conflicting timing model is introduced.
- Unresolved alignments are rare and explicitly marked.

## Validation

- Spot-check aligned events near section boundaries.
- Confirm bar numbering is 1-indexed and monotonic.
- Confirm density-per-bar output uses the aligned bar IDs rather than inferred ranges.

## Failure Cases

- Forced snapping across large timing offsets
- Section assignment lags behind actual section windows
- Alignment fields overwrite raw timing data

## Out of Scope

- Human correction UI for alignment
- Swing-aware or expressive timing transcription models
