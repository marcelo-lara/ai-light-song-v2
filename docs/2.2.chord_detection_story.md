# EPIC 2 - Story 2.2

## Chord Detection

## Goal

Infer chord events and confidence scores from beat-aligned harmonic features.

## Inputs

- HPCP features from Story 2.1
- Canonical beat and bar grid from Story 1.2

## Dependencies

- Template matching for chord likelihoods
- HMM or Viterbi decoding for temporal smoothing

## Processing Steps

1. Convert HPCP vectors into chord likelihoods using a defined chord template library.
2. Decode the per-beat chord likelihood sequence using a temporal model.
3. Merge stable runs into chord events with start time, end time, bar, and beat anchors.
4. Compute confidence values for both per-beat and merged event outputs.
5. Persist chord events into the harmonic layer contract.

## Output Artifacts

- chord event output used by `layer_a_harmonic.json`
- optional intermediate chord probability file

## Output Schema

```json
{
  "chords": [
    {
      "time": 10.91,
      "end_s": 14.29,
      "bar": 5,
      "beat": 1,
      "chord": "Fm",
      "confidence": 0.82
    }
  ]
}
```

## Acceptance Criteria

- Chord progression is musically plausible for test songs.
- Event segmentation follows the bar and beat grid.
- Confidence values reflect ambiguity rather than always clustering at one range.

## Validation

- Compare decoded chord sequences against validation-only reference chord files when they are available.
- Reference chord files are optional, read-only validation inputs and must never overwrite inferred chord outputs.
- Check repeated patterns for internal consistency.
- Confirm merged events cover the analyzed song range without gaps caused by decoder artifacts.

## Failure Cases

- Over-fragmented chord events
- Decoder collapses ambiguous regions into a single dominant chord
- Confidence values are not calibrated enough to express uncertainty

## Out of Scope

- CRNN phase for learned chord recognition
- Borrowed-chord music theory labeling beyond the chosen template vocabulary
