# EPIC 3 - Story 3.4

## Section Hint Inference

## Goal

Generate concise, human-editable section hints from the aligned symbolic timeline so downstream prompt-based workflows have stable structural guidance before LLM-friendly abstraction prose is built.

This story is limited to symbolic and structural hints that can be inferred from the post-3.3 aligned timeline. Drop, explode, chilldown, accent-callout, and other energy-driven hint types are explicitly out of scope here and belong to Story 5.4 song identifier inference.

## Inputs

- Symbolic summaries from Story 3.2
- Aligned phrase and section information from Story 3.3
- Canonical section windows from Story 4.2

## Dependencies

- Deterministic phrase-window grouping
- Stable section identifiers from `section_segmentation/sections.json`
- A merge policy that preserves user-authored hints while refreshing inference-authored hints

## Processing Steps

1. Read the aligned symbolic layer and canonical section windows.
2. Build deterministic per-section inference hints from section summaries, phrase windows, and repeated phrase-group or motif references.
3. Write the producer-scoped inferred payload to `data/artifacts/<Song - Artist>/symbolic_transcription/hints.json`.
4. Read `data/output/<Song - Artist>/hints.json` when it already exists.
5. Preserve every existing hint whose `source` is `user`.
6. Discard every existing hint whose `source` is `inference`.
7. Merge the preserved user hints with the newly inferred hints and write the editable result back to `data/output/<Song - Artist>/hints.json`.
8. Keep section identity explicit in the output by storing canonical `section_id`, `label`, `start`, and `end` alongside the hint list.

## Output Artifacts

- `data/artifacts/<Song - Artist>/symbolic_transcription/hints.json`
- `data/output/<Song - Artist>/hints.json`

## Acceptance Criteria

- The inferred artifact is producer-scoped and deterministic.
- The editable output file lives under `data/output/<Song - Artist>/`.
- Re-running the pipeline refreshes `source: inference` hints while leaving `source: user` hints untouched.
- Every output section row includes canonical `section_id`, `label`, `start`, `end`, and `hints`.
- Hint text is concise, human-readable, and LLM-friendly without inventing unsupported musical claims.
- The story does not infer drop, explode, chilldown, hit, or rise-style hints from symbolic data.

## Validation

- Confirm the output file is stable for repeated runs on the same song when no user edits were made.
- Confirm user-authored hints survive a rerun byte-for-byte within each section.
- Confirm repeated section labels remain distinguishable because placement is keyed by `section_id` rather than label text.
- Confirm every inferred hint is traceable to symbolic summaries, phrase windows, or motif repetition data.

## Failure Cases

- The implementation overwrites or deletes user-authored hints during regeneration.
- The output omits section identity fields and can only be matched by display label.
- The inferred hints claim transient or energy-stage facts not available from the symbolic timeline.
- The lighting score consumes `sections.json.hints` instead of the dedicated `hints.json` contract.

## Out of Scope

- Energy-derived drop or accent inference
- Editing UI beyond the raw JSON file
- Fixture-specific cue generation
