# EPIC 6 - Story 6.5

## Fixture-Aware Orchestration and Lighting Score Generation

## Goal

Convert abstract lighting behaviors into fixture-aware event plans and a fully populated `lighting_score.md` based on the rig definition.

The score must preserve exact deterministic cue times while also encoding how repeated motifs and harmonic patterns should repeat, vary, or escalate across the show.

## Inputs

- fixture-agnostic lighting events from Story 6.4
- unified analysis context from Story 6.3 when needed for explanation text
- fixture definitions from `data/fixtures/fixtures.json`
- the stable score contract in `docs/lighting_score_template.md`

The upstream inputs must already include exact section, phrase, accent, and pattern-occurrence times. Story 6.5 must not approximate those timings in prose.

## Dependencies

- Rig-role classification logic
- Fixture capability mapping
- Score population rules

## Processing Steps

1. Load the fixture configuration and classify fixtures into stable roles.
2. Translate abstract scene and behavior instructions into fixture-specific event assignments.
3. Respect fixture type, location, and base channel constraints.
4. Generate DMX-ready or DMX-near event data.
5. Populate `lighting_score.md` using the canonical template structure, preserving exact timing anchors and repeated-pattern or motif callback rules.

## Fixture Role Contract

The sample rig supports at least these derived roles:

| Role | Example fixtures |
| --- | --- |
| moving_head_main | `head_el150` |
| moving_head_fx | `mini_beam_prism_l`, `mini_beam_prism_r` |

## Output Artifacts

- fixture-aware `lighting_events.json` or equivalent
- `data/output/<Song - Artist>/lighting_score.md`

## Output Schema

```json
{
  "lighting_events": [
    {
      "time": 60.0,
      "fixtures": {
        "head_el150": {"pan": 120, "tilt": 45, "dimmer": 0.85},
        "parcan_l": {"dimmer": 0.60, "color": "blue"}
      }
    }
  ]
}
```

## Lighting Score Requirements

The generated score must populate these sections:

- Title using `# <Song Title> - Lighting Score`
- Metadata
- Feature Summary
- Timing Anchors
- Visual Strategy
- Fixture Intentions
- Section Plan
- Song-Specific Rules

## Acceptance Criteria

- Fixture roles are respected.
- The output score is readable by both developers and lighting designers.
- Section transitions and accents remain visible after fixture-specific mapping.
- Exact timing anchors remain explicit enough for an LLM or operator to place cues without guessing.
- Repeated motifs and harmonic patterns are translated into coherent callback or variation rules.
- Chorus-like high-energy passages read as more intense than intro or release passages.

## Validation

- Confirm the fixture-role classification matches `data/fixtures/fixtures.json`.
- Compare the generated score structure with the canonical template contract in `docs/lighting_score_template.md`.
- Confirm the score title uses the required `# <Song Title> - Lighting Score` format.
- Confirm all score timestamps match the structured timing anchors from Stories 6.3 and 6.4.
- Confirm repeated motif and pattern instructions stay faithful to the symbolic and Layer D artifacts.
- Verify that no fixture receives controls it cannot support.

## Failure Cases

- Fixture-aware mapping ignores physical layout or role symmetry
- Score prose rounds, shifts, or invents cue times that are absent from the deterministic artifacts
- Score generation produces generic text disconnected from the analyzed song
- Fixture-specific constraints are applied inconsistently across repeated sections

## Out of Scope

- Real-time lighting control transport
- Human-in-the-loop live desk editing workflow
