# EPIC 5 - Story 5.5

## Confidence, Review, and Override Workflow

## Goal

Expose event confidence, ambiguity, and editable review outputs so humans can confirm, correct, or replace machine-detected events without modifying raw analysis artifacts.

These review-facing files are validation-scoped human-review documents. They must not expand the stable UI output contract under `data/output/<Song - Artist>/`.

## Inputs

- event vocabulary and schema from Story 5.1
- machine-detected events from Story 5.4
- optional identifier hints from Story 4.5

## Dependencies

- stable confidence band definitions
- explicit candidate ranking for ambiguous windows
- separate storage for machine output and human-authored overrides

## Processing Steps

1. Define confidence bands such as high, medium, and low, and document their numeric thresholds.
2. Flag ambiguous windows where multiple event types or subtypes score similarly.
3. Export review-friendly machine output in JSON or markdown without requiring edits to raw analysis artifacts.
4. Support confirm, delete, retime, relabel, and note-taking operations in a separate override file.
5. Preserve provenance so the final export can distinguish machine-only events from human-confirmed or human-edited events.

## Output Artifacts

- `data/artifacts/<Song - Artist>/validation/song_events.review.json`
- `data/artifacts/<Song - Artist>/validation/song_events.review.md`
- `data/artifacts/<Song - Artist>/validation/song_events.overrides.json`

## Acceptance Criteria

- Ambiguous detections are visible instead of being silently forced into one label.
- Reviewers can confirm, delete, retime, relabel, and annotate events without editing raw analysis files.
- Machine and human-authored event versions remain distinguishable and reproducible.
- The review workflow does not add files to or remove files from `data/output/<Song - Artist>/`.

## Validation

- Confirm override files can be applied deterministically to the machine output.
- Confirm low-confidence windows preserve ranked alternatives and scores.
- Confirm the final merged event list records whether each event was machine-only, human-confirmed, or human-edited.

## Failure Cases

- Review output loses the original machine evidence or timing.
- Human edits overwrite machine output in place with no provenance.
- Low-confidence windows omit the candidate labels that a reviewer needs.

## Out of Scope

- Interactive GUI design
- ML model training
- Fixture-aware lighting orchestration

# EPIC 5 - Story 5.5

## Event Benchmarking and Genre-Sensitive Tuning

## Goal

Validate the event system on benchmark songs and tune configurable thresholds or profiles without changing the canonical event schema.

## Inputs

- reviewed machine or merged event outputs from Story 5.5
- benchmark song annotations maintained for this repository
- optional producer-scoped genre guidance from Story 6.1

## Dependencies

- stable timing tolerance rules for event matching
- documented genre-profile presets
- evaluation reporting that explains misses, false positives, and confusion patterns

## Processing Steps

1. Create or maintain benchmark annotations for the selected reference songs.
2. Compare machine or merged event outputs against the benchmark annotations using documented label and timing tolerances.
3. Record misses, false positives, timing offsets, and label confusions in a validation report.
4. Tune optional genre-sensitive threshold presets such as festival EDM, electro pop, alt rock, dark electronic, and downtempo.
5. Keep genre tuning optional so the generic event pipeline remains valid even without metadata.

## Output Artifacts

- `benchmark_annotations/`
- `data/artifacts/<Song - Artist>/validation/event_benchmark.json`
- `src/analyzer/contracts/event_threshold_profiles.json`

## Acceptance Criteria

- Benchmark annotations exist for the selected validation songs.
- The event pipeline can generate comparable outputs for those songs and report where it succeeds or fails.
- Genre-sensitive tuning improves detection quality without changing the schema or canonical vocabulary.

## Validation

- Confirm evaluation reports show label agreement, timing tolerance, false positives, and misses.
- Confirm generic detection still works when no genre profile is selected.
- Confirm profile selection remains deterministic for repeated runs on the same song and metadata.

## Failure Cases

- Benchmark comparisons collapse all failures into one aggregate score with no actionable detail.
- Genre tuning changes the schema or canonical event names.
- The pipeline requires a genre profile to run at all.

## Out of Scope

- Fixture-specific lighting behavior
- Learned classifiers
- Manual score authoring
