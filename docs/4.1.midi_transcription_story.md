# EPIC 4 - Story 4.1

## MIDI-like Transcription

## Goal

Transcribe all available stems and the full mix into note events, then validate and merge those note streams into a final symbolic artifact aligned to the canonical timing grid.

## Inputs

- Harmonic stem from Story 1.1
- Bass stem from Story 1.1
- Vocals stem from Story 1.1
- Drums stem from Story 1.1
- Source song mix from `data/songs/<Song - Artist>.mp3`
- Canonical beat grid from Story 1.2
- Section windows from Story 3.2 when section labels are attached during generation

## Dependencies

- Basic Pitch or equivalent transcription engine

## Phase 1 Default Implementation

- Use Spotify Basic Pitch across all available stems plus the full mix.
- Preserve raw Basic Pitch note outputs per source instead of collapsing them away so later stories can reuse pitch-bend and model-output details.
- Run a symbolic validation/finalization phase after transcription to score each source and decide which sources are promoted into the final note-event stream.
- Emit a final `layer_b_symbolic.json` note-event stream aligned to the canonical beat grid and current section windows.
- Keep per-source provenance explicit so downstream symbolic logic can distinguish bass-driven, harmonic, vocal, drum, and full-mix note evidence.
- Save raw per-source MIDI caches alongside JSON note caches for inspection and future symbolic tooling.

## Processing Steps

1. Run transcription on every available stem and on the full source mix.
2. Normalize the raw note-event outputs into a shared schema.
3. Preserve source provenance so later stories can distinguish bass-driven notes from harmonic, vocal, drum, and full-mix notes when needed.
4. Align each source stream to the canonical timing grid.
5. Validate each source stream using source-aware quality checks.
6. Promote the validated sources into a final note-event collection.
7. Attach timing, beat, bar, and section references.
8. Store `aligned_beat` as beat-in-bar and keep any absolute beat reference in a separate field such as `aligned_beat_global`.

## Output Artifacts

- `layer_b_symbolic.json`
- optional raw transcription cache for each analyzed stem

## Output Schema

```json
{
  "note_events": [
    {
      "time": 1.230,
      "end_s": 1.430,
      "duration": 0.200,
      "pitch": 64,
      "velocity": 0.50,
      "confidence": 0.80,
      "source_stem": "harmonic",
      "transcription_engine": "basic-pitch",
      "pitch_bend_range": {"min": -1, "max": 1},
      "pitch_bend_step_count": 4,
      "aligned_beat": 3,
      "aligned_beat_global": 3,
      "aligned_bar": 1,
      "section_name": "Intro"
    }
  ]
}
```

## Acceptance Criteria

- Main harmonic material is represented in the harmonic-stem transcription.
- Bass movement is represented clearly enough to support downstream bass-motion logic.
- Additional stem and full-mix transcriptions are validated before they are promoted into the final note-event stream.
- The merged event stream is usable by feature engineering without further schema cleanup.
- `aligned_beat` uses beat-in-bar semantics and does not overwrite raw onset time.

## Validation

- Spot-check note density against audible activity in sample songs.
- Confirm bass-stem notes cluster lower than harmonic-stem notes in pitch.
- Compare phrase behavior against reference structure where useful.

## Failure Cases

- Dense polyphony causes noisy note overgeneration
- Bass notes are missed or octave-shifted excessively
- Confidence values are missing or not comparable across stems

## Out of Scope

- Perfect symbolic transcription
- Human-edited MIDI cleanup
